T-01 · GitFlow y protección de ramas

Sem 1-2
ObjetivoEstablecer un flujo Git único en la org: trabajo en ramas feature/*, integración vía PR, ramas protegidas y despliegue solo desde main (tags).
HerramientasGitHub org Babooni-Technologies · Git CLI · guía GitFlow en README de cada subkit (react/front, react/back, angular/front, angular/back)
EntornoPiloto 1: repos react/back + react/front · Piloto 2: repos angular/back + angular/front · replicar plantilla al resto tras validar ambos
Guía paso a paso
A · Definir y crear el modelo de ramas
  1. Ramas permanentes: main (producción, siempre desplegable) y develop (integración). Si el repo solo tiene main, créala:
    git checkout main git pull origin main git checkout -b develop git push -u origin develop
  2. Rama por defecto en GitHub: Repo → Settings → General → Default branch → dejar main como producción. develop es la base de features.
  3. Convención de nombres:
    • feature/JIRA-123-descripcion-corta — nueva funcionalidad
    • fix/JIRA-456-bug-login — corrección sobre develop
    • hotfix/JIRA-789-critico — urgente desde main
    • release/1.4.0 — preparación de versión (opcional)
  4. Flujo diario del desarrollador:
    git checkout develop git pull origin develop git checkout -b feature/JIRA-123-mi-tarea # ... commits ... git push -u origin feature/JIRA-123-mi-tarea # Abrir PR en GitHub: base = develop, compare = feature/JIRA-123-mi-tarea
  5. Flujo a producción: PR develop → main (o release branch) → merge → tag semver → deploy automático (T-03).
    git checkout main git pull origin main git merge --no-ff develop git tag -a v1.2.0 -m "Release 1.2.0" git push origin main --tags
B · Proteger la rama main en GitHub
  1. Ir a: Repo → Settings → Branches (o Rules → Rulesets si usáis rulesets de org).
  2. Clic en Add branch protection rule (o New ruleset → target main).
  3. En Branch name pattern: escribir main.
  4. Activar estas opciones:
    • ☑ Require a pull request before merging
    • ☑ Require approvals: 1
    • ☑ Dismiss stale pull request approvals when new commits are pushed
    • ☑ Require status checks to pass before merging (configurar checks tras T-02: build, test)
    • ☑ Require branches to be up to date before merging
    • ☑ Do not allow bypassing the above settings
    • ☑ Restrict who can push to matching branches → solo bots/admins si aplica
    • ☑ Block force pushes
    • ☑ Do not allow deletions
  5. Guardar la regla. Probar: intentar git push origin main directo → debe rechazarse.
C · Proteger develop (reglas más ligeras)
  1. Repetir en Settings → Branches con pattern develop.
  2. Activar: PR obligatorio + 1 approval + status checks + block force push. No hace falta restringir push tan estricto como en main.
D · Plantilla de PR y flujo en todos los repos
  1. Crear archivo .github/pull_request_template.md en el repo piloto:
    ## Qué cambia - ## Cómo probar 1. ## Checklist - [ ] Tests pasan en CI - [ ] Sin secretos en el diff - [ ] Commits siguen convención (T-05)
  2. Repo → Settings → General → marcar Allow squash merging (recomendado) y desactivar merge commits si queréis historial limpio.
  3. Replicar reglas de protección en el resto de repos: Org → Settings → Rulesets → crear ruleset org-wide para main y aplicarlo a todos los repos activos.
E · Documentar en ai-foundation
  1. Añadir docs/gitflow.md en cada subkit (react/front/docs/, react/back/docs/, angular/front/docs/, angular/back/docs/) y enlace en el README del subkit. Los repos que ya copiaron el subkit reciben la guía con el próximo sync/PR.
  2. Actualizar tabla de stacks en ai-foundation/README.md (convención de ramas común a Java, Node, React y Angular).
  3. PR de ejemplo: abrir una feature trivial (ej. añadir el enlace al README) y mergear siguiendo el flujo completo como demostración al equipo.
Verificación: Push directo a main bloqueado · PR requiere 1 review · PR no mergeable si CI falla (tras T-02) · develop existe y está protegida · doc publicada en ai-foundation · al menos 1 PR real mergeado con el flujo.

T-02 · Pipeline CI (build + test)

Sem 1-2
ObjetivoPipeline en cada PR/push a main. Workflows en react/front, react/back, angular/front y angular/back.
HerramientasGitHub Actions · Java 11/17 + Gradle · Node LTS + npm · React (Vite/CRA) · Angular CLI · JUnit / Jest / Karma
EntornoRunners hosted GitHub · Piloto 1: react/back + react/front (sem 1-2) · Piloto 2: angular/back + angular/front (sem 3-4)
Guía paso a paso
A · Workflows dentro de cada subkit (no en carpeta suelta)
  1. Patrón del repo: el CI vive en {kit}/{front|back}/.github/workflows/. Al copiar la subcarpeta al proyecto, el workflow llega a .github/workflows/ del repo destino.
  2. React (front) — react/front/.github/workflows/ci.yml (complementa twin-check CI existente):
    name: React CI on: pull_request: { branches: [main, develop] } push: { branches: [main, develop] } jobs: build-and-test: name: build + test runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: { node-version: '20', cache: 'npm' } - run: npm ci - run: npm run build - run: npm test -- --watchAll=false --passWithNoTests
  3. Angular (front) — angular/front/.github/workflows/ci.yml:
    name: Angular CI on: pull_request: { branches: [main, develop] } push: { branches: [main, develop] } jobs: build-and-test: name: build + test runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: { node-version: '20', cache: 'npm' } - run: npm ci - run: npm run build -- --configuration=production - run: npm run test -- --watch=false --browsers=ChromeHeadless
  4. Java (back) — angular/back/.github/workflows/ci.yml:
    name: Java CI on: pull_request: { branches: [main, develop] } push: { branches: [main, develop] } jobs: build-and-test: name: build + test runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-java@v4 with: { distribution: temurin, java-version: '17', cache: gradle } - run: chmod +x gradlew - run: ./gradlew clean build -x test - run: ./gradlew test
  5. Node (back) — react/back/.github/workflows/ci.yml:
    name: Node CI on: pull_request: { branches: [main, develop] } push: { branches: [main, develop] } jobs: build-and-test: name: build + test runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: { node-version: '20', cache: 'npm' } - run: npm ci - run: npm run build --if-present - run: npm test -- --passWithNoTests
  6. Actualizar tabla de stacks en README.md del repo. Adoptar: copiar kit completo al proyecto (no solo el YAML suelto).
B · Instalar CI en repos piloto
  1. Piloto 1 — Frontend React: copiar react/front/ al repo piloto → PR feature/T-02-ci-frontend → Actions verde.
  2. Piloto 1 — Backend Node: copiar react/back/ al repo piloto API → PR feature/T-02-ci-node → Actions verde.
  3. Si no hay tests, añadir smoke test mínimo antes de exigir CI en branch protection.
  4. Mejoras del piloto 1 → PR de vuelta al subkit en ai-foundation + entrada en CHANGELOG.md.
  5. Piloto 2 — Frontend Angular: copiar angular/front/ al repo piloto → PR feature/T-02-ci-angular → Actions verde.
  6. Piloto 2 — Backend Java: copiar angular/back/ al repo piloto → PR feature/T-02-ci-java → ./gradlew test verde.
C · Enlazar CI con branch protection (T-01)
  1. Ejecutar el workflow al menos 1 vez con éxito en un PR.
  2. Repo → Settings → Branches → editar regla de main y develop.
  3. En Require status checks to pass, buscar y marcar: build + test (nombre exacto del job).
  4. Guardar. Verificar: PR con test roto no permite merge; botón Merge deshabilitado.
D · Cache y optimización por stack
  1. Java: cache: gradle en setup-java · Node/React/Angular: cache: npm en setup-node.
  2. Objetivo: pipeline < 10 min. Tests e2e pesados → job nightly en {kit}/{front|back}/.github/workflows/nightly-e2e.yml.
Verificación: Workflows en los cuatro subkits · CI verde en piloto 1 (react/back + react/front) y piloto 2 (angular/back + angular/front) · README tabla actualizada · CHANGELOG con versión del subkit.

T-03 · Despliegue automático PaaS (piloto)

Sem 1-2
ObjetivoDeploy automático a staging al mergear en main: APIs Java/Node en contenedor · frontends React/Angular como estático.
PrerrequisitosT-02 CI verde en repos piloto · cuenta/org PaaS creada · entorno staging activo (proyecto + servicios + BD si aplica) · secretos en GitHub/PaaS
HerramientasRailway / Render / Fly.io · Docker · workflows deploy en {kit}/{front|back}/.github/workflows/deploy.yml
EntornoPiloto 1: staging react/back + react/front (sem 1-2) · Piloto 2: staging angular/back + angular/front (sem 3-4) · prod solo tras validar staging + rollback en ambos pilotos
Orden lógico: primero tener un entorno staging operativo en el PaaS (cuenta, proyecto, BD, variables). Después conectar repos y automatizar el deploy. No al revés.
Guía paso a paso (ejemplo Railway — adaptar si usáis Render/Fly)
A · Entorno staging activo en el PaaS (antes de conectar repos)
  1. Elegir proveedor (Railway / Render / Fly.io) y crear cuenta de org Babooni (facturación, acceso del equipo).
  2. Crear proyecto staging (ej. babooni-staging) separado de producción futura.
  3. Si la API usa BD: provisionar instancia staging en el PaaS o apuntar a BD demo existente · anotar host, puerto, credenciales.
  4. Definir convención de URLs staging (ej. api-piloto-staging.railway.app, app-piloto-staging.onrender.com).
  5. Crear secretos antes del primer deploy:
    • En PaaS: DB_URL, DB_USER, DB_PASSWORD, PORT, SPRING_PROFILES_ACTIVE=demo
    • En GitHub (si deploy vía Actions): RAILWAY_TOKEN / equivalente del proveedor
  6. Verificar que el entorno responde: servicio vacío o hello-world desplegado manualmente → URL accesible · DNS/HTTPS OK.
  7. Documentar en {kit}/{front|back}/docs/runbook-deploy.md: URL staging, variables, responsable, enlace al proyecto PaaS.
B · Conectar repos piloto al staging (deploy manual primero)
  1. Precondición: T-02 CI verde en el repo piloto (build + test pasan).
  2. PaaS → New Service → conectar repo GitHub piloto 1 (react/back, luego react/front).
  3. Rama main como trigger de deploy automático (activar tras el primer deploy manual OK).
  4. Primer deploy manual → comprobar health (/health, /actuator/health o ruta propia).
  5. Repetir para piloto 2 (angular/back, angular/front) cuando piloto 1 esté estable.
  6. No commitear .env — añadir a .gitignore.
C · Backend — Node (piloto 1) y Java (piloto 2)
  1. Piloto 1 — API Node (react/back): variables ya en PaaS (fase A). Build: npm run build + node dist/main.js · Dockerfile node:20-alpine.
  2. Piloto 2 — API Java (angular/back): repetir flujo tras validar piloto 1. server.port=${PORT:8080} · Dockerfile o Jib:
    FROM eclipse-temurin:17-jre-alpine WORKDIR /app COPY build/libs/*.jar app.jar EXPOSE 8080 ENTRYPOINT ["java","-jar","/app.jar"]
  3. Plantillas deploy: react/back/.github/workflows/deploy.yml (Node) · angular/back/… (Java) · frontends en fase B.
D · Frontend — React (piloto 1) y Angular (piloto 2)
  1. Piloto 1 — React (react/front): build estático npm run build → dist/.
  2. Piloto 2 — Angular (angular/front): ng build --configuration=production → dist/nombre-app/.
  3. En PaaS (Render Static Site / Railway / Netlify interno): root directory = carpeta de build · SPA fallback /* → index.html.
  4. Variables frontend en PaaS: API_BASE_URL (URL del backend staging de fase B) · IDP_CLIENT_ID si aplica.
  5. Plantilla CI deploy en cada subkit frontend (react/front/.github/workflows/deploy-static.yml, angular/front/…).
E · Deploy desde GitHub Actions (opcional, tras staging manual OK)
  1. Generar token de deploy en el PaaS (Railway: Project → Settings → Tokens).
  2. GitHub repo → Settings → Secrets → Actions → crear RAILWAY_TOKEN (o equivalente).
  3. Añadir job al workflow, solo en push a main y tras CI verde:
    deploy: needs: build-and-test if: github.ref == 'refs/heads/main' && github.event_name == 'push' runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-java@v4 with: { distribution: temurin, java-version: '17', cache: gradle } - run: ./gradlew bootJar - name: Deploy to Railway uses: berviantoleo/railway-deploy@v1 with: railway_token: ${{ secrets.RAILWAY_TOKEN }} service: mi-servicio-piloto
F · Rollback y criterio de paso a prod
  1. En el PaaS: Deployments → deploy anterior → Rollback · probar en staging.
  2. Actualizar runbook con URL staging, variables y procedimiento de rollback por stack.
  3. Prod: solo cuando staging piloto 1 (y luego 2) llevan ≥1 semana estable · rollback probado · CI + deploy automático verificados.
Verificación: Entorno staging activo y documentado · deploy manual OK en piloto 1 (y piloto 2) · health check · rollback probado · merge a main despliega a staging · secretos solo en PaaS/GitHub Secrets.

T-04 · SonarQube / SonarCloud + Quality Gate

Sem 3-4
ObjetivoAnálisis estático en CI con Quality Gate común: cobertura ≥70% en código nuevo, 0 vulnerabilidades nuevas, deuda controlada.
HerramientasSonarCloud · Gradle (Java) · sonar-scanner (React/Angular/Node) · SONAR_TOKEN
EntornoSonarCloud org Babooni · Piloto 1: react/back + react/front · Piloto 2: angular/back + angular/front
Guía paso a paso
A · Alta en SonarCloud y conexión GitHub
  1. Ir a sonarcloud.io → login con cuenta GitHub de la org.
  2. + → Analyze new project → importar org Babooni-Technologies.
  3. Conectar repos piloto 1 (react/back, react/front) → Set Up → GitHub Actions. Repetir con repos piloto 2 (angular/back, angular/front) tras validar piloto 1.
  4. Copiar el SONAR_TOKEN que muestra SonarCloud.
  5. GitHub repo → Settings → Secrets → Actions → SONAR_TOKEN = token copiado.
  6. Org GitHub → Settings → Secrets → Actions → mismo secret a nivel org si varios repos lo usarán.
B · Java — Gradle + JaCoCo (piloto 2)
  1. En build.gradle, añadir plugin:
    plugins { id 'org.sonarqube' version '5.1.0.4882' } sonar { properties { property 'sonar.projectKey', 'Babooni-Technologies_nombre-repo' property 'sonar.organization', 'babooni-technologies' property 'sonar.host.url', 'https://sonarcloud.io' } }
  2. Generar reporte JaCoCo para cobertura (si no existe):
    plugins { id 'jacoco' } test { finalizedBy jacocoTestReport } jacocoTestReport { dependsOn test }
  3. Local: ./gradlew test jacocoTestReport sonar -Dsonar.token=TU_TOKEN para validar antes de CI.
  4. Snippet en subkit Java: angular/back/config/sonar.gradle.snippet (copiar a build.gradle del proyecto).
C · React, Node, Angular — sonar-scanner (piloto 1 y piloto 2)
  1. Piloto 1: conectar Sonar en repos react/back y react/front. Piloto 2: repetir en angular/back y angular/front.
  2. Crear sonar-project.properties en raíz del repo frontend/API:
    sonar.projectKey=Babooni-Technologies_nombre-repo sonar.organization=babooni-technologies sonar.sources=src sonar.tests=src sonar.javascript.lcov.reportPaths=coverage/lcov.info sonar.sourceEncoding=UTF-8
  3. React/Angular: generar cobertura en CI antes del scan — npm test -- --coverage (Jest/Vitest) o ng test --code-coverage.
  4. Step Sonar en workflow del subkit: {kit}/{front|back}/.github/workflows/sonar.yml o step añadido a ci.yml:
    - uses: SonarSource/sonarqube-scan-action@v4 env: SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }} GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  5. ESLint/Prettier ya en react/front/ y angular/front/; alinear Sonar con esas configs al copiar el subkit.
D · Step Sonar en GitHub Actions (Java — piloto 2)
  1. Añadir al workflow ci.yml tras el job de test:
    - name: SonarCloud Scan env: SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }} GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }} run: ./gradlew jacocoTestReport sonar
  2. Push → verificar análisis en dashboard SonarCloud del repo.
E · Quality Gate org-wide
  1. SonarCloud → Quality Gates → duplicar «Sonar way» → nombrar Babooni Standard.
  2. Condiciones mínimas:
    • Coverage on New Code ≥ 70%
    • New Issues (Vulnerabilities) = 0
    • Technical Debt Ratio on New Code ≤ 5%
  3. Administration → Quality Gate → asignar Babooni Standard como default de la org.
  4. GitHub → branch protection → añadir status check SonarCloud Code Analysis como obligatorio.
  5. Publicar criterios en docs/quality-gate.md (raíz ai-foundation; enlazar desde README de cada subkit).
Verificación: SonarCloud verde en piloto 1 y piloto 2 · QG pasa en PR limpio · PR con vulnerabilidad o cobertura baja bloquea merge · documento quality-gate publicado.

T-05 · Convención de commits

Sem 3-4
ObjetivoCommits legibles y automatizables: Conventional Commits en todos los repos, validados en CI.
HerramientasCommit Message Guide · commitlint · GitHub Action
EntornoLocal (hook opcional) + validación en PR
Guía paso a paso
A · Formato Conventional Commits
  1. Formato: <tipo>(<alcance>): <descripción>
  2. Tipos permitidos: feat, fix, docs, style, refactor, test, chore, ci, build, perf.
  3. Ejemplos válidos:
    feat(auth): add JWT refresh token endpoint fix(invoice): correct VAT calculation rounding docs(readme): add local setup instructions ci(github): add SonarCloud scan step
  4. Publicar guía en cada subkit: {kit}/{front|back}/docs/commit-convention.md + commitlint.config.js en la raíz del subkit (se copia al proyecto).
B · Validación en CI (recomendado)
  1. Crear .github/workflows/commitlint.yml:
    name: Commitlint on: pull_request: types: [opened, synchronize, reopened, edited] jobs: commitlint: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 with: { fetch-depth: 0 } - uses: wagoid/commitlint-github-action@v6
  2. Crear commitlint.config.js en raíz:
    module.exports = { extends: ['@commitlint/config-conventional'], rules: { 'subject-case': [0], 'header-max-length': [2, 'always', 100] } };
  3. PR con commit mal formado (ej. fixed stuff) → workflow falla → corregir con git commit --amend o rebase.
C · Adopción en repos piloto
  1. Piloto 1: activar commitlint en repos react/back y react/front.
  2. Piloto 2: activar commitlint en repos angular/back y angular/front.
  3. Añadir sección en CONTRIBUTING.md de cada repo con enlace al subkit origen en ai-foundation.
  4. Comunicar al equipo: squash merge usa el título del PR como commit en main — el título del PR también debe seguir la convención.
  5. Plantilla PR (T-01): añadir ítem «Título del PR sigue Conventional Commits».
Verificación: Doc en ai-foundation · CONTRIBUTING actualizado · Action commitlint verde en PR válido y rojo en inválido · equipo informado.

T-06 · Arquetipos por stack + arquitectura + dependencias

Sem 3-4
ObjetivoUn arquetipo ejecutable por tecnología dentro de {kit}/{front|back}/archetype/, más documentación de capas, ADRs transversales y dependencias compartidas.
HerramientasSpring Boot / Express o Fastify · Vite · Angular CLI · Gradle/npm · GitHub Template repos · GitHub Packages · ADRs en docs/adr/
EntornoArquetipos extraídos de piloto 1 (react/back + react/front) y piloto 2 (angular/back + angular/front)
Guía paso a paso
A · Kit vs arquetipo (no confundir)
  1. Subkit (react/front/, react/back/…): reglas IA, CI, configs — se copia a repos existentes.
  2. Arquetipo ({kit}/{front|back}/archetype/): proyecto mínimo arrancable — se usa para repos nuevos.
  3. Cada arquetipo incluye: código de ejemplo, test smoke, /health, Dockerfile, workflows del subkit y {kit}/{front|back}/archetype/docs/architecture.md.
  4. Convenciones de capas (documentar en architecture.md de cada kit):
    • Java: controller → service → dao · DTOs en API, nunca entidad JPA expuesta.
    • Node: routes → services → repositories · validación en capa de entrada.
    • React: pages → components → hooks → services/api · estado global solo si hace falta.
    • Angular: features → smart/dumb components → services · lazy loading por feature.
  5. ADRs transversales (org-wide) en raíz de ai-foundation:
    docs/adr/ ├── template.md ├── 001-capas-vs-microservicios.md └── 002-migracion-idp.md
B · Arquetipos ejecutables (rutas en ai-foundation)
  1. Mapa de arquetipos por tecnología:
    react/back/archetype/ ← API Node react/front/archetype/ ← app React angular/back/archetype/ ← API Java (Spring Boot) angular/front/archetype/ ← app Angular
  2. Estructura común de cada arquetipo:
    {kit}/{front|back}/archetype/ ├── README.md ← «Cómo crear repo desde este arquetipo» ├── docs/architecture.md ├── src/ ← código mínimo funcional ├── tests/ ← al menos 1 test smoke ├── Dockerfile ├── .github/workflows/ ← copiados del subkit (ci.yml, deploy.yml) └── .cursor/ ← reglas IA del subkit
  3. Node (back, piloto 1) — react/back/archetype/ — extraer del piloto T-02/T-03:
    src/ ├── app.js ← bootstrap Express/Fastify ├── routes/ │ └── health.routes.js ← GET /health → 200 ├── services/ │ └── example.service.js ├── repositories/ ← vacío o mock de ejemplo └── middleware/ └── errorHandler.js tests/ └── health.test.js ← supertest: GET /health
  4. React (front, piloto 1) — react/front/archetype/ (Vite + TypeScript):
