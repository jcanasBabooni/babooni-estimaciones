Esta guia describe el procedimiento que Babooni sigue para elaborar estimaciones de proyectos cerrados de desarrollo software. El objetivo es dejar una propuesta defendible, trazable y sin huecos graves, lista para revision interna y posterior envio al cliente por parte de Direccion.
Punto de partida

A · Funcional del cliente

El cliente entrega el funcional. Babooni lo versiona y pasa a la fase 2 (analisis de dudas).

  • Identificar version (v.1, v.2...)
  • Convertir a Markdown si hace falta
  • Continuar a fase 2 obligatoriamente

B · Funcional elaborado por Babooni

Babooni genera el analisis a partir de briefings o material parcial. En ese proceso ya se resuelven la mayoria de dudas.

  • Versionar el documento
  • Mantener en Markdown como fuente de trabajo
  • La fase 2 suele no aplicar o ser residual

Como crear el funcional Babooni

  • Plantilla y guia para crear el funcional Babooni (Google Drive)

Si el funcional viene del cliente, se adapta o completa hasta cumplir esa plantilla antes de estimar.

Formato de trabajo

El funcional se mantiene en Markdown (.md) para maximizar el contexto con IA y facilitar iteraciones. A partir de el se generan los HTML de entrega cuando corresponda.

Cuando aplica / cuando no

Aplica: funcional del cliente con ambiguedades o dependencias no cerradas.
No aplica (o residual): funcional generado por Babooni, con dudas ya resueltas en el analisis.

  • Leer el funcional completo y marcar ambiguedades, contradicciones y supuestos
  • Separar dudas de negocio y dudas tecnicas
  • Identificar preguntas criticas (las que mas mueven horas)
  • Preparar y enviar documento(s) de informacion a solicitar
  • Registrar hipotesis temporales si hay que avanzar sin respuesta

Entregables tipicos

  • Informacion_a_solicitar_negocio_*.html
  • Informacion_a_solicitar_equipo_tecnico_*.html
  • Respuestas de negocio y/o tecnico recibidas y clasificadas
  • Funcional actualizado (nueva version) con las decisiones
  • CU y CA revisados tras el feedback
  • Alcance entra / no entra alineado
  • Dudas abiertas restantes explicitadas para la tabla de riesgos

Criterio de salida

Funcional versionado con las secciones obligatorias (incluye CU + CA). Siguiente paso: diseno UX/UI.

Que es y que no es

Es diseno: mockups, wireframes, Figma, criterios UX/UI, estados de pantalla.
No es diseno: implementacion FE en codigo (va en la estimacion, fase 5, al mismo nivel que el Backend).

Quien lo hace en Babooni

Babooni cuenta con disenador UX/UI que se encarga del diseno en la mayoria de proyectos. En algunos clientes el diseno lo aporta el cliente; en ese caso Babooni revisa y valida contra el funcional, sin elaborarlo.

  • Pantallas y flujos del funcional traducidos a mockups / Figma
  • Estados clave cubiertos (vacio, carga, error, exito, permisos)
  • Modales, formularios, tablas y navegacion alineados con CU
  • Decisiones UX que mueven alcance documentadas y validadas
  • Enlace o export del diseno disponible como input de la estimacion FE

Caso habitual · Diseno Babooni

El disenador Babooni produce mockups/Figma a partir del funcional cerrado. Las horas de diseno se incluyen en la estimacion como bloque UX/UI, separado de la implementacion FE.

Excepcion · Diseno del cliente

El cliente aporta Figma/mockups. Babooni los revisa contra el funcional; si hay huecos, se abren dudas antes de estimar el FE. No se imputan horas de elaboracion de diseno.

Criterio de salida

Diseno UX/UI cerrado: mockups / Figma validados, sin pantallas criticas pendientes ni decisiones UX abiertas que muevan alcance. Solo entonces se pasa a estimar (fase 5).

Prerequisito: diseno cerrado

La estimacion se elabora con el diseno UX/UI cerrado (fase 4 completada). No se estima la implementacion FE sobre mockups provisionales, incompletos o con decisiones UX pendientes. Si el diseno no esta cerrado, se vuelve a la fase 4 (o se abren dudas) antes de continuar.

Fuentes obligatorias

  • Plantilla de estimaciones Babooni (Google Drive)
  • Funcional en Markdown (secciones completas, con CU y CA)
  • Diseno UX/UI cerrado: mockups / Figma validados (fase 4)

Checklist de construccion

  • Tareas Backend desglosadas con horas
  • Diseno UX/UI cerrado verificado antes de desglosar horas FE
  • Tareas de implementacion Frontend desglosadas a partir del diseno cerrado (mockups/Figma)
  • Si Babooni elabora el diseno: bloque UX/UI separado del FE
  • Actividades transversales incluidas
  • Cronograma por sprints coherente con el alcance (sin forzar un tope artificial de sprints)
  • Riesgos con delta de horas si aplica
  • Premisas y exclusiones explicitas
  • Trazabilidad CU/CA + mockups → bloques de horas
  • HTML de estimacion con estandar visual Babooni
  • Buffer +20% aplicado antes de enviar a Direccion

Buffer obligatorio +20%

Sobre el total de horas de la estimacion tecnica base (tras coordinacion y desglose). El documento a Direccion ya lleva el buffer incluido y el cronograma debe ser coherente con ese total.

Condicionada vs en firme

En firme (objetivo): funcional cerrado + diseno UX/UI cerrado + lo critico de negocio/tecnico resuelto.
Condicionada: solo por riesgos residuales explicitos; no por diseno incompleto. Sin diseno cerrado no hay estimacion de implementacion FE.

  • Revision linea a linea de tareas, horas, premisas y riesgos
  • Suma BE + FE (+ UX/UI si aplica) + transversal cuadra
  • Implementacion FE estimada a partir de mockups/Figma
  • Buffer +20% verificado
  • Cronograma coherente con capacidad y total majorado
  • Alineada con funcional + diseno
  • Calidad estandar incluida (tests BE/FE, plan PRO)
  • HTML revisado (ortografia, cifras, logos)
  • Segunda persona Babooni ha revisado cuando sea posible

No dar por lista si...

  • No hay funcional o version clara
  • Faltan CU o CA (u otras secciones obligatorias criticas)
  • Alcance entra / no entra indefinido
  • El diseno UX/UI no esta cerrado (mockups/Figma incompletos o decisiones UX abiertas)
  • Horas o cronograma incoherentes
  • Falta el buffer +20%
  • Estimacion entregada a Direccion con el +20% ya aplicado
  • Direccion envia al cliente coste + horas / tiempo de desarrollo
  • No se desarrolla nada hasta aceptacion formal del cliente
  • Aceptacion registrada antes de planificar arranque

Responsabilidad de Direccion

El equipo elabora la estimacion tecnica. Direccion Babooni traslada coste y calendario comercial al cliente.

Gate: estimacion aceptada

1. Funcional tecnico (HTML)

  • Arquitectura y repos, referencias de codigo
  • Flujo funcional-tecnico, contratos API, modelo de datos
  • Validaciones BE/FE, tareas diarias, plan por sprints, CA
  • Funcional_Tecnico_<NombreProyecto>.html

2. QA Checklist Backend (HTML)

  • QA_Checklist_<NombreProyecto>_Backend.html

3. QA Checklist Frontend (HTML)

  • QA_Checklist_<NombreProyecto>_Frontend.html

Orden post-aprobacion

  1. Cerrar funcional tecnico (funcional + mockups + estimacion)
  2. Derivar checklists QA BE y FE
  3. Arrancar desarrollo con estos artefactos disponibles

Regla de oro

No arrancar desarrollo sin aceptacion formal de la estimacion. Dudas abiertas: explicitas en funcional y en riesgos de la estimacion.
