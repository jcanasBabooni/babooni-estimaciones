1.1 Servicio actual de justificante (Historial Medico) CRITICO

Modulo, microservicio y repositorio donde vive hoy la opcion Ficha trabajador > Historial medico > Justificante de asistencia

Funcionalidad partida entre dos repositorios: UI y controladores en eanaliticas; logica de negocio + DAO en eanaliticas-model-jdk8. No hay microservicio independiente, se despliega como un unico WAR en Tomcat.

  • Vista (JSP): eanalitica-wm/web/WEB-INF/jsp/historialMedico/jspf/Reconocimiento.jspf:862 — el enlace se muestra solo si reconocimiento.justificante == true. Funciones JS: descargarJustificante(rmId) y enviarJustificante(rmId)
  • Controladores: JustificanteController.java → justificante.do?action=descargarRM (genera y descarga PDF) y justificante.do?action=enviarRM (envia por correo). Controlador alternativo: DescargarJustificanteReconocimientoController.java → descargar-justificante-reconocimiento.do
  • Servicio: spring-service.xml:172 → JustificanteReconocimientoManager
  • DAO: spring-dao.xml:159 → SpringJdbcJustificanteReconocimientoDao
Contrato completo del endpoint o servicio actual: metodos, parametros, respuesta, codigos de error

Confirmado en codigo legacy (eanaliticas): no es REST; son acciones Spring MVC sobre el WAR eanaliticas.

  • Descarga PDF: GET {baseUrl}justificante.do?action=descargarRM&rmId={rmId} — respuesta binaria application/x-file-download, nombre Justificante.pdf. Genera PDF en caliente, lo envia al stream y borra el fichero temporal. Sin JSON de error estructurado.
  • Envio email: POST {baseUrl}justificante.do?action=enviarRM&rmId={rmId} — respuesta JSON SuccessBean (icono: OK/AVISO, mensajes[]). Genera PDF, adjunta en Base64 y llama communication-api con tipo=1421, evento=1421 (TIPO_ENVIAR_JUSTIFICANTE_RM).
  • Alternativa descarga: descargar-justificante-reconocimiento.do (DescargarJustificanteReconocimientoController).
  • Parametro unico: rmId (integer). Sesion SSO requerida (TokenSesion en sesion HTTP).
  • Visibilidad UI: enlace solo si reconocimiento.justificante == true (Reconocimiento.jspf).

Pendiente negocio/BD: criterio que marca justificante=true en el objeto reconocimiento (no esta en el Manager del justificante).

Como se calculan hoy las horas de entrada y salida automaticamente

Logica en: JustificanteReconocimientoManager.java metodo generarDocumento() (l.284–353) y SpringJdbcJustificanteReconocimientoDao.java.

Hora de entrada — 5 casos de precedencia en getFechaInicio() (l.98–129):

  1. Si hay cita y FCH_ADMISION cae el mismo dia que HORA_FIN y es anterior a RM_FCH_ALTA → se usa FCH_ADMISION
  2. Idem si la admision es posterior al alta pero es extraccion previa (extraccion_previa = 1) → se usa igualmente FCH_ADMISION
  3. Si hay cita pero no es extraccion previa → se usa RM_FCH_ALTA
  4. Si hay cita y es extraccion previa → fecha del traspaso de la extraccion previa al reconocimiento (log INF4_RM_LOG estado 33)
  5. Si no se cumple ninguno → HORA_INICIO de PROV_RUTAS_CITAS

Hora de salida — getFechaFinMedEnf() (l.131–150): ultimo estado 27 (fin consulta medica) y ultimo estado 28 (fin enfermeria) en INF4_RM_LOG. Se toma la marca temporal mas tardia.

Ajuste Canarias: si el centro esta en Las Palmas (cod. 35) o Santa Cruz de Tenerife (cod. 38), ambas horas se restan una hora.

Si hoy ya existe plantilla PDF o motor de generacion reutilizable

Existen plantillas ODT versionadas en el repositorio templates, carpeta vs/reconocimientos/, con una version por idioma: es, ca, eu, en, gl, fr, de (JustificanteReconocimiento.odt).

Motor actual: JOOReports + JODConverter sobre OpenOffice/LibreOffice. Recomendacion: migrar a generacion basada en informes de reconocimientos para eliminar la dependencia de OpenOffice.

Que datos del reconocimiento / cita / trabajador consume el servicio actual

Nombre completo del trabajador, NIF, fecha del reconocimiento, nombre de la empresa, puesto de trabajo, puestos adicionales, hora de entrada, hora de salida y fecha de emision (fecha del dia en que se genera el justificante).

Si el flujo actual registra algo en log o solo deja constancia de envio

El justificante no deja trazabilidad propia en base de datos. El unico rastro es el que genera communication-api al registrar el envio del correo. El servicio de eanaliticas no escribe ni consulta INF4_RM_LOG ni ninguna tabla de auditoria propia.

Si el flujo actual usa el modulo de comunicaciones o envia de otra forma

Usa communication-api, el modulo centralizado de comunicaciones de la plataforma.

Que porcentaje del desarrollo puede ser refactor / extension frente a desarrollo nuevo

Estimacion orientativa tras analisis de codigo: ~40–50% portacion de SQL y logica de horas/datos desde legacy a planificacion-backend; ~50% desarrollo nuevo (API Minerva, plantilla Thymeleaf, adaptador delegacion HM, SMS temporal).
