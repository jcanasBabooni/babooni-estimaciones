Propuesta PaaS · Azure

Entorno PaaS para la infra de Ideauto · Aprovechando Azure ya contratado

Propuesta Babooni → Ideauto
Propuesta de adoptar un modelo PaaS (Platform as a Service) sobre la suscripcion Azure que Ideauto ya tiene contratada. Objetivo: estandarizar despliegues, secretos y operacion, reduciendo el riesgo operativo frente a despliegues manuales.
Contexto

Por que PaaS en Azure

Ideauto ya dispone de Azure. Un enfoque PaaS (App Service / Container Apps, pipelines, Identity, Key Vault o secretos de pipeline) permite que la plataforma gestione runtime, escalado, parches, HTTPS, identidades y despliegue continuo, mientras el equipo se centra en la aplicacion.

Es una buena solucion porque industrializa lo que hoy depende de pasos manuales: menos variabilidad entre entornos, menos dependencia de personas concretas y un camino mas claro hacia controles de seguridad (secretos fuera del codigo, identidades gestionadas, trazabilidad de release).

Despliegues manuales

Errores que se evitan

Con pipelines + PaaS se reducen o eliminan fallos tipicos del despliegue a mano:

  • Olvidar un fichero / artefacto o desplegar una build incorrecta / desactualizada.
  • Copiar mal configuraciones entre pre y prod (URL, connection strings, flags).
  • Dejar secretos en codigo o en scripts locales (passwords, JWT, API keys).
  • Desalineacion de versiones entre front, API y dependencias.
  • Pasos omitidos (migraciones BD, reinicio de servicios, invalidacion de cache, certificados).
  • Entornos “a mano” no reproducibles (funciona en un servidor y no en otro).
  • Rollback dificil o imposible cuando un release falla.
  • Falta de auditoria: quien desplego, cuando y con que commit.
  • Errores humanos en IIS / carpetas / permisos al publicar a mano.
  • Ventanas de downtime mas largas por intervenciones manuales.
Ventajas

PaaS / Azure

  • Despliegue repetible y automatizado (CI/CD).
  • Secretos gestionados (pipelines y/o Key Vault + Managed Identity).
  • Escalado, health checks y reinicios gestionados.
  • HTTPS / certificados y parches de plataforma.
  • Menor carga operativa para Ideauto y Babooni.
  • Mejor alineacion con auditoria / pentest (trazabilidad).
Siguiente paso

Que pedimos a Ideauto

  • Validar la adopcion de un entorno PaaS sobre Azure existente.
  • Otorgar permisos de pipelines a Babooni (alternativa viable a Key Vault puro).
  • Definir entornos (dev / pre / prod) y ownership de Resource Group.
  • Acordar el modelo de secretos (Pipeline secrets vs Azure Key Vault + Managed Identity).
