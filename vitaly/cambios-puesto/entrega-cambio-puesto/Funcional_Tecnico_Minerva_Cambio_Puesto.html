Objetivo del evolutivo (RF v.6)

Desde la ficha de un reconocimiento pendiente, permitir cambiar cliente, centro, puesto principal y puestos adicionales; recalcular impacto con la misma logica que la creacion del RM (puestos + ACP + periodicidad/fecha recomendada); mostrar ventana resumen solo con deltas; y al confirmar actualizar RM, tab Resultados, RL automaticamente y log CA-17. No tocar el flujo desde Historial Medico (CA-16).

UX Flujo usuario (solo RM pendiente + ACL)

  1. Usuario autorizado ve icono "cambiar puesto" junto al puesto en ficha RM (CA-01)
  2. Abre modal: busca cliente (≥3 chars o CIF) → centros → puesto principal → puestos adicionales
  3. Si hay comunicacion lab activa → aviso CA-15 informativo (icono + texto; no bloquea Continuar)
  4. Pulsa Continuar → BE calcula impacto (dry-run) → ventana resumen solo deltas + nota fija pruebas cumplimentadas
  5. Cancelar en resumen → no se modifica nada (CA-12)
  6. Confirmar cambios → transaccion BE: UPDATE INF_RM + RL + log CA-17 + refresh FE resumen/Resultados
  7. Si nuevos parametros/perfiles analiticos con analitica ya comunicada → warnings CU-11 (Resumen + Comunicar lab)

BE Orquestacion al confirmar (rrmm-backend)

  1. Validar: estado Pendiente (8), ACL, centro con VS, RL activa en destino, centro no V1
  2. Resolver rlId: PPT_CEN_TRA activa (FCH_BAJA IS NULL) por trabajador/cliente/centro
  3. Aplicar escenario RL (tabla seccion 4.3 entrega Preving): editar / crear / asociar / bloquear
  4. Llamar POST puestos-back /puestos/{rlId}/cambiar-puesto cuando proceda
  5. UPDATE INF_RM (puesto, cliente, centro, periodicidad, flags lab, audit trail…)
  6. Efectos colaterales en misma transaccion: analiticas, log RM si cambia cliente, validacion RM…
  7. INSERT INF4_RM_LOG_CAMBIO_PUESTO (CA-17)
  8. Post-commit: reasignarRMCita() si cambia centro

§6.2 Flujo alternativo — actualizar protocolos sin cambiar puesto

RF v.6 permite recalcular protocolos/ACP sin modificar cliente/centro/puesto. Misma ventana resumen y confirmacion transaccional, pero el UPDATE INF_RM no cambia campos de puesto/cliente/centro — solo impacto en protocolos, periodicidad y fecha. Implementar en BE-06/BE-07 + FE-12. RL no se toca si puesto/cliente/centro no cambian.
